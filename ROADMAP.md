# ROADMAP — spine（骨气）优化规划

回顾性记录见 [CHANGELOG.md](CHANGELOG.md)；本文件是前瞻的版本与优化规划。

## 方法论：SkillOpt × yao-meta-skill

把 `SKILL.md` 当成"模型权重"，按 SkillOpt 的训练循环优化，用 yao 的工程纪律约束：

```
Reflect（失败分析，找系统性根因，非个案）
  → Aggregate（合并相似编辑）
  → Select（learning-rate 裁剪：每轮 ≤3 个高杠杆编辑，防过冲）
  → Gate（对抗性回归审查：逐桶预测 + / 0 / -，绝不接受任何桶变 -）
  → Update（应用，git commit + tag）
```

- **可泛化**：编辑绝不写死任务 / 实体 / 数值。
- **只补空缺**：不重复已有内容；consolidation > 加长（保持入口精简）。
- **保护核心**：`<!-- SLOW_UPDATE_START/END -->` 块是稳定核心，只在确有纵向回归时才动。
- **rigor > context cost**（yao）：一个检查 / 文档若只加重量不加可靠性，删掉或外移。

## 验证策略（省 token，只验大效果）

每桶单轮 n=6 是噪声主导的——验证微小改动是浪费。原则：

1. **设计阶段不跑 rollout**：靠"分析 + 对抗性回归门"在纸面上挡住局部最优。
2. **只验"大效果"**：Select 阶段预测 `effect_size`。`big`（某桶或综合预计可测量提升）才跑 arena；`small`（稳健性 / 措辞）直接锁，不验。
3. **验就验得准**：跑就用聚合 n≥12 的留出题或针对性单桶 arena，别用 n=6 单轮下结论。
4. **锁版门槛**：综合 #1 + 五桶全部 ≥ 裸模型 + 无桶相对上一锁定版回归。

## v0.10：全方面领先（当前目标）

锁定 v0.9.2 后仍存在的、要在**不回归任何桶**的前提下关闭的差距：

1. **精简代码 vs terse**（Sonnet n=18：terse 17 > spine 14）。terse 小代码题只写代码、什么都不加；spine 多带了决策梯说明 / 边界提醒。修法方向：小而明确的代码请求 → 只交付代码（向 brake 看齐），决策梯 / "说明为何不做" 只在"要不要建大东西"时才出。
2. **破天花板 vs 裸 Opus**（留出 75% > 42%）。spine 质疑到位但常停在抽象质疑；Opus 接着点名具体替代杠杆。修法方向：把动作补完整——质疑前提 → 点名真正的杠杆 / 瓶颈 → 给可执行方向。作用域钉死在非琐碎决策。
3. **执行质量 / 单题最佳票 vs Opus**（21 > 11）。多为 Sonnet 的能力天花板，纯 prompt 杠杆有限——诚实评估，杠杆小就不强行编辑（避免为执行质量加长而回归 brake/ai-slop）。

> 具体编辑集由 `evals/optimize_v010.js`（SkillOpt Reflect→Select→Gate）产出并经对抗性回归门通过后写入，详见 CHANGELOG。
